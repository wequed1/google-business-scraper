## 先搞清楚：你要抓的到底是什么数据

很多人把"Google Business Profile"和"Google Maps"混着说，其实它们是同一份数据的两个入口。Google Business Profile 是商家自己管理的资料页（店名、地址、电话、营业时间、评分、评论、"从企业获悉"的简介等），这些信息会同时展示在 Google 搜索的本地结果框和 Google 地图上。

所以无论你搜的是"google business profile scraper"还是"google maps scraper"，最终要拿到的字段基本是同一套：

- 店铺名称、分类标签（比如"意大利餐厅"还会细分出"利古里亚餐厅""那不勒斯餐厅"这种子分类）
- 完整地址 + 经纬度坐标
- 电话号码、官网链接
- 星级评分与评论数量（部分工具能拿到评论原文）
- 营业时间，包括"高峰时段"这种动态数据
- 价格区间标签（$、$$、$$$）
- 商家配图

## 抓取 Google Business Profile 数据合法吗？

这是搜索这个关键词的人最常问的问题，必须老老实实说清楚。公开可用的商业数据抓取在大多数情况下是合法的，美国第九巡回上诉法院在 hiQ Labs v. LinkedIn 案中明确表示，抓取公开数据不违反《计算机欺诈与滥用法》（CFAA），这一判例基本奠定了当前美国网络抓取合法性的基础。具体到 Google 地图这类商家数据——店名、电话、地址、评分、评论数——这类数据通常被认为属于该判例覆盖的范畴。

不过有两点要注意：

> 第一，**只抓公开数据，别碰登录墙后面的内容**。第二，**欧洲用户要留意 GDPR**，抓取商家这类 B2B 联系方式（比如餐厅的公开电话）通常可以归入"合法利益"条款，但如果你的目标是评论区里具体某个用户的姓名，那已经涉及个人数据，处理方式要更谨慎。

ScraperAPI 官方在自己的 FAQ 里也明确表态：抓取地理位置信息是合法的，只要你不绕过任何登录限制，ScraperAPI 不会抓取登录墙后面的数据，确保你的抓取项目完全合规。这等于把"别碰登录墙"这条红线也写进了产品设计里。

## 为什么直接用脚本"裸抓" Google 总是抓两天就被封

如果你试过自己写 Python 脚本直接请求 Google 地图或者 Google 本地搜索结果页，应该已经体会过这种痛苦：开发自己的 Google 地图抓取工具如果没有几年经验积累会相当有挑战性，你需要应对来自 Google 的一连串防护手段，包括 IP 保护（代理）、Cookie 与会话管理、浏览器模拟、页面结构频繁变化等等。更现实的问题是，Google 地图的结果是滚动加载出来的，普通请求根本拿不到完整列表，得用 Puppeteer/Selenium 这类工具模拟真实用户滚动行为，而这套裸抓的脚本极容易被反爬机制识别，IP 随时可能被封。

这也是为什么市面上的"GMB Scraper"基本分成两大类：

| 工具类型 | 典型代表 | 适合谁 | 主要短板 |
|---|---|---|---|
| 浏览器插件 / 桌面软件 | Chrome 插件类 GMB 工具、Windows 客户端 | 偶尔抓几百条线索的个人用户 | 量一大就容易被限速或封号，无法做定时任务 |
| 无代码云端平台 | 各类 SaaS 抓取平台 | 不想写代码、要的是现成 CSV | 灵活度低，遇到非标准页面结构容易抓空 |
| 开发者向的抓取 API | ScraperAPI 等 | 需要把抓取嵌进自己系统、做定时/大批量采集的团队 | 需要一点点代码基础（但通常几行就够） |

如果你的需求只是"这周末抓 200 家本地咖啡馆的电话"，插件或无代码工具完全够用。但如果你的场景是**持续监控某个行业在多个城市的商家数据**、**给整个销售团队搭建线索库**、**每周自动刷新一次本地排名情况**，裸抓脚本和轻量插件大概率撑不住，这时候才是 API 方案该登场的时候。
